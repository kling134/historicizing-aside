Historicizing <aside>

Introduction

In her 1999 article On the Relationship Between Old and New Technologies, Christina Haas argues against oversimplified views of technological development. According to Haas, the so-called simple replacement model and the straightforward progress model “are based on the premise that old and new technologies are separate, distinct, mutually exclusive, and easily differentiated” (p. 210). However, Haas contends that “technological change cannot be a simple replacement of one technology with another…because technology is anchored so closely both to individual histories and to cultural practices” (p. 212). As a result, some people will continue to use “old” technologies as long as those technologies continue to suit their purposes.

Furthermore, older technologies often serve as the inspiration for the development of newer technologies, and as a result, it is not uncommon for newer technologies to incorporate some of the same conventions associated with their older counterparts. For instance, many features that originated with print media can also be seen in electronic media contexts. The HTML element known as <aside> is but one example of this phenomenon.

What is <aside>?

HTML, or Hypertext Markup Language, is the code that webpages are made of. The units of content within an HTML document are known as elements.

Since its inception in the early 1990s, HTML has undergone many changes. The specification for the latest iteration, HTML5, included several new elements, one of which was <aside>. According to the HTML5 specification, “The aside element represents a section of a page that consists of content that is tangentially related to the content around the aside element, and which could be considered separate from that content.”

Another way to understand what the <aside> element means is to reflect on the definition of the word itself—according to the Merriam-Webster dictionary, an aside is “a comment or discussion that does not relate directly to the main subject being discussed.” Thus, content wrapped in <aside> tags can be said to not relate directly to the main content of the page.

Special features

The <aside> element belongs to the sectioning content model, which means that a new section is automatically created in the document outline when the <aside> element is employed. The other elements belonging to the sectioning content model are <article>, <nav>, and <section>. Each of the four sectioning elements was new to HTML5.

In addition, the <aside> element is noteworthy for its accessibility. <aside> is a semantic element, which means that it describes its contents. One benefit of semantic markup is that it allows users of assistive technologies, such as screen readers for the visually impaired, to easily navigate from one content area to another.

According to the Mozilla Developer Network, “Accessible Rich Internet Applications (ARIA) defines ways to make Web content and Web applications…more accessible to people with disabilities.” ARIA consists of role, state, and property attributes that can be added to markup language such as HTML. The <aside> element carries a default ARIA role attribute value of 'complementary,' which means that user agents such as browsers and screen readers will treat an <aside> element as a “supporting section of the document, designed to be complementary to the main content.”

The development of <aside>

Before HTML5, there was only one block-level element that allowed web developers to divide pages into different areas: <div>. This generic container can be used to group content and apply CSS styles.

Through the magic of CSS, web developers can style <div> elements so that they appear to be distinct page sections. However, if you remove the styling, it becomes apparent that <div> elements are really just part of the linear flow of the document. Indeed, that is exactly how the browser computes <div> elements, even when styling is present. As a result, users of assistive technologies may have a hard time accessing information on webpages that exclusively use the <div> element to group content.

The problem with <div> is that it carries no meaning on its own. Web developers can imbue it with some meaning by assigning class or id attributes that describe the element’s contents, but this is more for the developer’s benefit than anyone else’s—the browser can’t meaningfully differentiate between <div id="header"> and <div id="footer">. Moreover, web developers are free to use whatever class and id attributes they please, which means there is little consistency from one webpage to the next.

New structural elements

In 2007, the web standards body known as the World Wide Web Consortium (W3C) partnered with the Web Hypertext Application Technology Working Group (WHATWG) to form an HTML Working Group dedicated to drafting a new specification for HTML5. The editors recognized the need to introduce new structural elements to supplement <div>. They looked to the results of a large-scale analysis by Google to identify potential new elements.

In December 2005, Google had conducted an analysis of over one billion webpages in order to study web-authoring practices. Among other things, this analysis yielded 20 of the most popular class names used by developers, including several which would go on to become elements in their own right (e.g., footer, header, nav).

The element formerly known as <sidebar>

Interestingly, the <aside> element was known by another name for a time. According to front-end web developer Lachlan Hunt, “Originally, the [aside] element was named <sidebar>, and it was introduced to address the common case of left and right columns, of which one of the most common names classes/ids was ‘sidebar.’”

As the name suggests, the <sidebar> element was initially intended to be used for columns of content on the side of a page. However, at some point, the scope of the <sidebar> element expanded to include any “tangentially related content.” Once the intended use of the <sidebar> element evolved, it was only appropriate to change the name of the element to match its new meaning.

Ian Hickson, one of the editors who helped develop the HTML5 specification, favored the <sidebar> name change because he “wanted to avoid being too specific about the presentation in the name.” Whereas the <sidebar> element prescribed a format for displaying tangentially related content, the <aside> element leaves the matter of presentation up to the web developer.

How was <aside> shaped by print media?

The HTML5 specification makes reference to print media’s influence on the development of the <aside> element. It points out that tangentially related content is “often represented as sidebars in printed typography.” Indeed, open any newspaper, magazine, or textbook, and you are likely to encounter supplemental information of some kind displayed in a box off to the side. This convention carried over into webpage design, which undoubtedly set the stage for the introduction of the <aside> element.

Using <aside>

The HTML5 specification recommends that the <aside> element be used for “typographical effects like pull quotes or sidebars, for advertising, for groups of nav elements, and for other content that is considered separate from the main content of the page.” The specification also explicitly discourages developers from using the <aside> element for parentheticals “since those are part of the main flow of the document.”

Thus, <aside> can be used to achieve some of the same typographical effects seen in print media (e.g. pull quotes), and it can also be used for features unique to electronic media (e.g. related reading links).

Best practices

In keeping with the recommendations set forth in the HTML5 specification, web developers should consider using <aside> to encapsulate “tangentially related content.” This might include definitions, background information, or pull quotes, among other possibilities.

In addition, developer Mike Robinson writes that it is acceptable to use <aside> for “secondary content related to the site as a whole” so long as the <aside> element is not nested within an <article> element. In Robinson's own words, “When used within an article element, the contents should be specifically related to that article (e.g., a glossary). When used outside of an article element, the contents should be related to the site (e.g., a blogroll, groups of additional navigation, and even advertising if that content is related to the page).”

The <aside> element is still relatively new. As we have seen, some entities such as The Atlantic have already added the sectioning elements introduced in HTML5 to their web authoring toolkits. However, there are many other organizations that continue to rely on <div> to create the illusion of document sections. As more web developers come around to the idea of using <article>, <nav>, <section>, and <aside> in addition to <div>, new uses for each of these technologies may well come to light. It remains to be seen how the usage of the <aside> element will evolve in the future.
